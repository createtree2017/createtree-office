## 1. 프로젝트 개요

### 1.1 목적
- **사내 인수인계 매뉴얼** 관리 (리치 텍스트 에디터 기반)
- **업무(Task) 할당 및 추적** 시스템 (템플릿 기반 양식 응답)
- **거래처(병원) 관리** — 구글 드라이브 자료실 연동/계약 관리/서비스 계약
- **온라인 평판 모니터링** — 네이버/카카오/구글 리뷰 수집·AI 분석·자동 보고서
- **텔레그램 알림** — 모니터링 결과 자동 발송
- **직원 전용 포털** (RBAC 기반 접근 제어)

## 4. 인증 및 권한 시스템

### 4.1 JWT 인증 흐름
1. 클라이언트가 `POST /api/auth/login`으로 이메일/비밀번호 전송
2. 서버가 bcrypt로 비밀번호 검증 → JWT 토큰 발급 (7일 만료)
3. 토큰 페이로드: `{ id, email, role }`
4. 이후 모든 API 요청에 `Authorization: Bearer <token>` 헤더 필수

### 4.2 미들웨어

| 미들웨어 | 파일 | 기능 |
|----------|------|------|
| `authenticateToken` | `middleware/auth.ts` | JWT 검증 + DB에서 최신 사용자 정보 확인 (승인 여부 포함) |
| `authorizeRole(roles)` | `middleware/auth.ts` | 특정 역할만 접근 허용 (예: `["ADMIN"]`) |

### 4.3 역할(Role) 체계

| 역할 | 설명 | 주요 권한 |
|------|------|----------|

## 7. 모니터링 시스템 상세 — [신규 섹션]

### 7.1 아키텍처 개요

```
┌─────────────────────────────────────────────────────────────────┐
│                    모니터링 파이프라인                            │
│                                                                 │
│  수집(Collector)  →  AI 감성분석  →  보고서 생성  →  알림 발송    │
│                     (Gemini)       (HTML/Excel)    (Telegram)   │
└─────────────────────────────────────────────────────────────────┘
```

### 7.2 수집기(Collector) 분기 구조

| 플랫폼 | 수집 방법 | 담당 모듈 | 상태 |
|--------|----------|----------|------|
| 네이버 블로그/카페/뉴스 | 네이버 검색 API | `naverCollector.ts` | ✅ 정상 |
| 네이버 플레이스 | GraphQL API (pcmap-api) | `placeCrawler.ts` | ✅ 정상 |
| 카카오맵 | 비주얼 스크래핑 (Playwright + Gemini Vision) | `visionCrawler.ts` | ✅ 정상 |
| 구글 플레이스 | Outscraper SaaS API | `googlePlaceCollector.ts` | ✅ 정상 |

### 7.3 서비스 모듈별 역할

| 파일 | 줄 수 | 역할 |
|------|------|------|
| `monitoringService.ts` | ~330줄 | 워크플로우 오케스트레이터 (수집→분석→저장→알림) |
| `visionCrawler.ts` | ~600줄 | Playwright 스크린샷 + Gemini Vision 리뷰 추출 |
| `naverCollector.ts` | ~140줄 | 네이버 검색 API 수집기 |
| `placeCrawler.ts` | ~290줄 | 네이버 플레이스 GraphQL + 카카오맵 REST API |
| `googlePlaceCollector.ts` | ~160줄 | Outscraper API 구글 리뷰 수집 |
| `contentCrawler.ts` | ~120줄 | 게시글 본문 보강 크롤링 (cheerio) |
| `sentimentAnalyzer.ts` | ~200줄 | Gemini AI 감성 분석 + 키워드 폴백 |
| `reportGenerator.ts` | ~300줄 | HTML 보고서 + 구글 드라이브 업로드 + 엑셀 |
| `schedulerService.ts` | ~190줄 | setInterval 기반 cron 파싱 자동 실행 (30초 틱) |
| `types.ts` | ~130줄 | PostData, AnalysisResult 등 공유 타입 |

### 7.4 실행 흐름

```
POST /api/monitoring/templates/:id/execute
  1. DB에 결과 레코드 생성 (status: "RUNNING")
  2. 비동기로 processMonitoring() 실행
  3. templateType 분기:
     ├── 'place' → platform별 수집기 호출
     │   ├── naverplace → placeCrawler.crawlNaverPlace()
     │   ├── kakaomap  → visionCrawler.crawlByVision()
     │   └── googleplace → googlePlaceCollector.crawlGooglePlace() (Outscraper API)
     └── 'integrated' → naverCollector + contentCrawler
  4. sentimentAnalyzer.analyze(posts)  — Gemini AI 감성 분석
  5. reportGenerator.generateAndUpload() — HTML + 드라이브 업로드
  6. DB 결과 업데이트 (status: "COMPLETED")
  7. telegramService.sendMonitoringAlert() — 텔레그램 알림
```

### 7.5 구글 플레이스 수집 — 해결 완료

- **이전 이슈**: Playwright 비주얼 스크래핑 방식은 구글의 봇 감지(CAPTCHA, 동적 DOM 변경)로 인해 안정적 수집 불가
- **해결 방법**: **Outscraper SaaS API** 방식으로 전환하여 개발 완료 (`googlePlaceCollector.ts`)
- `monitoringService.ts`에서 `platform === 'googleplace'`일 때 Outscraper API 호출 경로로 라우팅
- `.env`의 `OUTSCRAPER_API_KEY` 사용 중

## 8. 텔레그램 알림 시스템 — [신규 섹션]

### 8.1 연동 흐름

```
1. 관리자가 거래처에 대한 텔레그램 초대 코드 생성
2. 초대 코드를 텔레그램 봇에 전송 (/start 코드)
3. 봇이 웹훅으로 서버에 전달 → chatId 저장 → 연동 완료
4. 모니터링 완료 시 해당 거래처의 chatId로 알림 발송
```

### 8.2 현재 상태
- 코드: ✅ 완전히 구현됨 (`telegramService.ts`, `notification.ts` 라우트)
- **배포 서버(Railway)**: ✅ Bot Token 설정 완료, **정상 작동 중**
- **로컬 개발 환경**: `.env`의 `TELEGRAM_BOT_TOKEN` 의도적으로 비어있음
  - 이유: 로컬과 배포 서버에서 동시에 토큰이 설정되면 웹훅 충돌로 터미널 에러 발생
  - 로컬에서 텔레그램 테스트 시 배포 서버 토큰을 임시 비활성화 필요

## 15. 핵심 비즈니스 로직 요약

### 15.1 업무 생성 → 드라이브 연동 흐름
```
업무 생성 (POST /api/tasks)
  ├── templateId가 있으면?
  │     ├── clientId가 있으면? → 거래처 드라이브 폴더 하위에 업무 폴더 생성
  │     └── clientId 없으면? → MASTER_ROOT_FOLDER에 업무 폴더 생성
  ├── 드라이브 폴더 ID를 tasks.driveFolderId에 저장
  └── 업무 DB 레코드 생성
```

### 15.2 거래처 라이프사이클 — [신규]
```
거래처 생성 (POST /api/clients)
  ├── 드라이브 폴더 자동 생성 (공유 드라이브 내)
  ├── 계약서 파일 첨부 (선택)
  └── DB 저장

서비스 계약 추가 (POST /api/client-contracts)
  ├── 거래처 드라이브 폴더 내 템플릿 서브폴더 생성
  └── clientServiceContracts 레코드 생성

계약종료 (POST /api/clients/:id/terminate)
  ├── 드라이브 폴더명 변경 ("종료_{이름}_{날짜}")
  ├── "계약종료" 폴더로 이동
  └── contractEndedAt 기록

거래처 완전삭제 (DELETE /api/clients/:id)
  ├── 계약종료 상태만 가능 (guard)
  ├── 드라이브 폴더 → 휴지통
  ├── users.clientId / tasks.clientId → NULL 처리 (FK 해제)
  └── DB 삭제 (clientServiceContracts CASCADE)
```

### 15.3 모니터링 실행 흐름 — [신규]
```
모니터링 실행 (POST /api/monitoring/templates/:id/execute)
  1. DB 결과 PENDING 생성 → RUNNING 전환
  2. templateType별 수집
     ├── place → platform별 수집기 호출 (네이버/카카오/구글)
     └── integrated → 네이버 API + 콘텐츠 보강
  3. Gemini 감성 분석 (일괄)
  4. HTML 보고서 생성 + 구글 드라이브 업로드
  5. 엑셀 데이터 생성
  6. DB 결과 COMPLETED 업데이트
  7. 텔레그램 알림 발송 (notifyEnabled)
```

### 15.4 텔레그램 연동 흐름 — [신규]
```
1. 관리자: POST /api/notification/telegram/invite/:clientId → 초대 코드 생성
2. 관리자: 초대 URL을 거래처에 전달
3. 거래처: 텔레그램에서 봇에 /start {CODE} 전송
4. 봇 웹훅: POST /api/notification/telegram/webhook
   → 코드 매칭 → clients.telegramChatId 저장 → 연동 완료 메시지
5. 이후 모니터링 완료 시 자동 알림 발송
```
