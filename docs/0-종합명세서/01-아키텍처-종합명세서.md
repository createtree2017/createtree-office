## 1. 프로젝트 개요

### 1.1 목적
- **사내 인수인계 매뉴얼** 관리 (리치 텍스트 에디터 기반)

users (1) ←── (N) manuals       [manuals.authorId → users.id]

users (1) ←── (N) taskTemplates [taskTemplates.authorId → users.id]

## 4. 인증 및 권한 시스템

### 4.1 JWT 인증 흐름
1. 클라이언트가 `POST /api/auth/login`으로 이메일/비밀번호 전송
2. 서버가 bcrypt로 비밀번호 검증 → JWT 토큰 발급 (7일 만료)
3. 토큰 페이로드: `{ id, email, role }`
4. 이후 모든 API 요청에 `Authorization: Bearer <token>` 헤더 필수

### 4.2 미들웨어

| 미들웨어 | 파일 | 기능 |
|----------|------|------|
| `authenticateToken` | `middleware/auth.ts` | JWT 검증 + DB에서 최신 사용자 정보 확인 (승인 여부 포함) |
| `authorizeRole(roles)` | `middleware/auth.ts` | 특정 역할만 접근 허용 (예: `["ADMIN"]`) |

### 4.3 역할(Role) 체계

| 역할 | 설명 | 주요 권한 |
|------|------|----------|
| **ADMIN** | 최고 관리자 | 전체 접근, 사용자/거래처 관리, 드라이브 전체 |
| **MANAGER** | 매니저 | 매뉴얼 작성/수정, 업무 관리, 드라이브 전체 |
| **HOSPITAL_ADMIN** | 거래처 관리자 | 자기 거래처 드라이브만 접근 |
| **USER** | 일반 (깡통) | 매뉴얼 읽기, 업무 응답만 가능 |

### 4.4 회원가입 규칙
- **첫 번째 사용자**: 자동으로 `ADMIN` + 자동 승인
- **이후 사용자**: `USER` 등급 + 자동 승인 (개발환경 설정. 운영 시에는 `isApproved: false`로 변경 필요)

## 12. 핵심 비즈니스 로직 요약

### 12.1 업무 생성 → 드라이브 연동 흐름
```
업무 생성 (POST /api/tasks)
  ├── templateId가 있으면?
  │     ├── clientId가 있으면? → 거래처 드라이브 폴더 하위에 업무 폴더 생성
  │     └── clientId 없으면? → MASTER_ROOT_FOLDER에 업무 폴더 생성
  ├── 드라이브 폴더 ID를 tasks.driveFolderId에 저장
  └── 업무 DB 레코드 생성
```

### 12.2 업무 응답 흐름
```
담당자가 TaskResponsePage 접속
  ├── GET /api/tasks/:id → 업무 + 템플릿(formSchema) 로드
  ├── GET /api/task-responses/:taskId → 기존 응답(임시저장) 로드
  ├── 작성 중 → PUT /:taskId/draft → 임시저장 (task → IN_PROGRESS)
  └── 완료 → POST /:taskId/submit → 최종 제출 (task → COMPLETED)
```

### 12.3 드라이브 검색 알고리즘
```
검색 요청 (GET /api/drive/search?q=검색어)
  1. RBAC 권한 체크 (USER 차단, HOSPITAL_ADMIN 격리)
  2. 시작 폴더 결정
  3. BFS로 하위 폴더 ID 전수 수집 (10개씩 청크 → 병렬 처리)
  4. 수집된 폴더 ID들에서 검색어 매칭 (15개씩 청크 → 병렬 처리)
  5. 중복 제거 → 정렬 (폴더 우선, 생성일자 역순)
```
