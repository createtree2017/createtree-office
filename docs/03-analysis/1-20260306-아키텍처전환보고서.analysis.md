# 아키텍처 마이그레이션 분석 보고서 (Google Drive Headless CMS)

- **작성일**: 2026-03-06
- **목적**: 현재 창조트리(React+Express+Neon) 시스템을 '구글 드라이브를 원본(SSOT)으로 삼는 Headless CMS 아키텍처'로 전환하기 위한 타당성 및 작업 내역 분석

## 1. 결론 요약

**"완전히 새로 만들 필요는 없으며, 기존 시스템 위에 '동기화 파이프라인(Sync Pipeline)'만 추가 구축하는 형태로 점진적 전환이 충분히 가능합니다."**

현재 창조트리 프로젝트는 **React 프론트엔드 - Express 백엔드 - Neon(PostgreSQL) DB**라는 매우 튼튼하고 유연한 모던 스택을 이미 갖추고 있습니다. 따라서 시스템을 갈아엎는 것이 아니라, 기존 백엔드에 구글 데이터를 받아오는 **'채널'만 뚫어주면 되는 작업**입니다.

## 2. 도입을 위해 필요한 추가 작업 내역

기존 코드를 버리지 않고 확장하는 형태로 아래 작업들이 필요합니다.

### A. 백엔드 (Express & Database) 작업 (핵심)
1. **Webhook 수신 API 개발**: 구글 앱스 스크립트(GAS)가 "데이터 변경됨" 신호를 보낼 때 이를 받을 라우터(ex. `POST /api/webhook/google-sync`) 생성.
2. **동기화 로직 구현**: 신호를 받으면 해당 구글 문서/시트의 ID를 이용해 내용을 가져와서 우리 네온 DB(`manuals` 테이블 등)에 `upsert` 하는 로직 개발.

### B. 프론트엔드 (React) 작업
1. **에디터 걷어내기**: 구글 드라이브가 메인 에디터가 되면, 무거운 `ManualEditor`(Tiptap) 대신 **"구글 문서 링크 연결"** 위주의 가벼운 UI로 관리자 페이지가 단순화됩니다.
2. **뷰어/폼 고도화**: 구글에서 넘어온 데이터나 응답 폼을 우리 사이트 디자인 시스템에 맞춰 아주 예쁘게 렌더링(보여주는 부분)하는 자체 React 컴포넌트(폼 태그 등)를 새로 만들어야 합니다. (iframe 제거)

### C. 구글 앱스 스크립트 (GAS) 작업
1. 구글 문서나 시트를 '수정'할 때마다 자동으로 실행되는 **트리거(Trigger)** 작성.
2. 수정된 내용을 우리 창조트리 서버의 Webhook 주소로 쏘아주는(HTTP POST) 코드 작성.

## 3. 마이그레이션 전략 (어떻게 진행할 것인가)

한 번에 모두 바꾸기보다는 점진적 도입을 권장합니다.

1. **Step 1. (현재 적용됨)**: 구글 폼 iframe 연동으로 구글-사이트 간 기본 연결고리 세팅.
2. **Step 2. 폼 데이터 단방향 동기화**: (가장 쉬운 백엔드 작업) 구글 폼에 접수된 데이터를 우리 네온 DB로 복사해 오는(Pull 또는 Webhook) 파이프라인부터 구축. (UI는 당장 안 바꿔도 됨)
3. **Step 3. 매뉴얼/텍스트 완전 전환**: '구글 문서'에서 작성한 글을 우리 테이블로 동기화(Sync)시키는 웹훅 및 UI 적용.
4. **Step 4. 자체 폼 UI로 전면 교체**: 기존 iframe 폼을 완전히 걷어내고, 우리 사이트만의 자체 폼(디자인 커스텀) 화면으로 전면 교체 (데이터는 구글과 네온 양방향 통신).

## 4. 총평

**"새로 만들 필요 전혀 없음. 현재 잘 다져진 React + Drizzle + Neon 기반 위에 '구글 데이터 수집용 파이프'만 몇 개 더 꽂는 훌륭한 확장 시나리오입니다."**

현재 스택이 위와 같은 고도화된 아키텍처 전환을 완벽하게 소화할 수 있습니다. 위 로드맵 중 **Step 2 (폼 데이터 단방향 동기화)**부터 당장 설계를 시작해 볼 수 있습니다.
