# Phase 3: 업무 관리 (Todo & Calendar) 설계서

## 1. 데이터 모델 설계 (Drizzle ORM)

### `tasks` 테이블
```typescript
export const taskStatusEnum = pgEnum("task_status", ["PENDING", "IN_PROGRESS", "ON_HOLD", "COMPLETED"]);

export const tasks = pgTable("tasks", {
  id: serial("id").primaryKey(),
  title: text("title").notNull(),
  description: text("description"),
  status: taskStatusEnum("status").default("PENDING").notNull(),
  dueDate: timestamp("due_date"),
  assigneeId: integer("assignee_id").references(() => users.id).notNull(), // 업무를 부여받은 사람 (필수)
  authorId: integer("author_id").references(() => users.id).notNull(),     // 업무를 부여한 사람 (필수)
  createdAt: timestamp("created_at").defaultNow().notNull(),
  updatedAt: timestamp("updated_at").defaultNow().notNull(),
});
```

## 2. API 엔드포인트 설계 (tasks.ts)

| Method | Endpoint | Description | Auth |
|--------|----------|-------------|------|
| GET | `/api/tasks` | 전체/내 업무 목록 조회 | Approved User |
| POST | `/api/tasks` | 새 업무 생성 | Approved User |
| PATCH | `/api/tasks/:id` | 업무 정보/상태 수정 | Author or Assignee |
| DELETE | `/api/tasks/:id` | 업무 삭제 | Author or Admin |

## 3. 프론트엔드 UI 구조

### 컴포넌트 구성
- `TasksPage`: `/tasks` 경로의 메인 페이지.
  - `TaskTabs`: '리스트 형태'와 '캘린더 형태' 전환.
  - `TodoBoard`: 상태별(대기/진행/완료) 카드 섹션.
  - `TaskCalendar`: 일정 기반 시각화 (마감일 기준).
- `CreateTaskModal`: 중앙집중식 모달 시스템에 등록될 생성/수정 폼.

### UI/UX 통일성
- 매뉴얼 시스템과 동일한 카드 스타일 (`bg-slate-900/50`, `border-slate-800`) 사용.
- 상태 변경 시 즉각적인 피드백 (Toast 알림).

## 4. 데이터 흐름
1. `useQuery`로 서버에서 전체 업무 데이터를 패칭.
2. Todo 리스트와 캘린더가 동일한 `queryKey`의 데이터를 공유하여 데이터 불일치 방지.
3. 업무 완료 체크 시 `useMutation`으로 즉시 서버 반영 및 캐시 갱신.
