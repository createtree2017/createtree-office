# 프로젝트 계획서: 구글 드라이브 기반 Headless CMS 도입

- **작성일**: 2026-03-06
- **상태**: 계획(Plan)
- **담당자**: AI 어시스턴트 & 관리자

## 1. 개요 및 목적
- **현재의 문제점**: 기존의 내부 DB 의존형 텍스트 에디터는 모바일이나 외부 환경에서의 작성/관리가 불편하며, 구글 폼 단순 임베드는 UI 통일성을 해치고 데이터를 활용(표시, 통계)하기 어렵습니다.
- **도입 목적**: 전 세계적으로 가장 훌륭한 문서(문서, 시트) 도구인 '구글 드라이브'를 메인 관리자 패널(SSOT)로 활용하고, 창조트리 사이트는 이 데이터를 빠르고 예쁘게 보여주는 단말기(Headless Viewer) 역할에 집중하여 **관리 편의성**과 **사이트 속도 및 안정성**을 극대화합니다.

## 2. 점진적 도입 범위 (Phases)

리스크를 최소화하기 위해 시스템을 한 번에 뒤집지 않고 점진적으로 파이프라인을 뚫어나갑니다.

- **Phase 1: 데이터 단방향 수집 (현재 최우선 목표)**
  - 내용: 구글 폼(구글 시트에 연동됨)으로 들어온 신청 데이터를 우리 네온 DB에 복사(Sync)합니다.
  - 효과: 구글 폼의 편리함을 누리면서, 신청자 데이터를 우리 사이트의 "마이페이지(신청내역)" 등에서 자유롭게 쓸 수 있게 됩니다.
- **Phase 2: 매뉴얼 텍스트 양방향 동기화**
  - 내용: 구글 문서(Docs)에서 작성한 내용을 사이트의 '매뉴얼' 상세 페이지 콘텐츠로 즉각 동기화합니다.
  - 효과: Tiptap 등 복잡한 에디터를 걷어내어 사이트가 가벼워집니다.
- **Phase 3: 프론트엔드 폼 자체 구축 (최종 완성)**
  - 내용: iframe(액자) 형태로 띄우던 구글 폼을 완전히 걷어내고 창조트리의 세련된 디자인으로 폼 UI를 새로 만듭니다. 제출은 우리 서버를 거쳐 구글 시트로 백업됩니다.

## 3. 리소스 및 제약 사항
- **Google Apps Script(GAS) 의존성**: 구글 측에서 API 변경이나 할당량(Quota) 제한이 발생할 수 있으므로, 주기적으로 데이터를 Bulk(대량) 동기화하는 안전장치 배치가 요구됩니다.
- **데이터 구조**: 구글 시트의 1열(Header) 이름이 DB 컬럼 이름과 정확히 매핑되는 룰(Rule) 확립이 필요합니다.

## 4. 진행 여부 확인 (Check)
- 관리자님의 승인이 떨어지는 즉시 **Phase 1(폼 데이터 단방향 동기화)** 설계를 시작하고 코딩에 돌입합니다.
